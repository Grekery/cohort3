Welcome

This is a repository for our capstone project. In this project we install GNURadio on the PS side of the ZedBoard and accelerate the process by moving computational intensive block, WBFM, into the PL side of the ZedBoard.

Contributors:

Instructions

In order to get this project working, there are a number of steps to take. The first is to get the XillyBus image working on your ZedBoard if you have a ZedBoard... if you don't then go out and buy one. How to install Xillybus on a SDCard and resize it

Because the ZedBoard has a limited amount of memory available, we are going to need to create swap space. When working with a large SD card (16 GB) with a good speed rating, we can write memory to the card by creating a swap file on it. How to make swap space in Ubuntu

It turns out the best way to install GNU Radio on the ZedBoard is not with package manager or with PyBOMBS, but to build it from source. This has the added bonus of allowing you to easily look through all the files on your system and easily modify them. How to install GNU Radio on ZedBoard

Now then, if you are using the RTL-SDR antenna (an extremely cheap transceiver), you'll need to install the drivers yourself. There's also a funny thing that Ubuntu does where it tries to take any digital antenna plugged in and automatically assign it to receiving digital TV, so there's a command you have to use in order to release it. You have to type this command every time unless you add it to a blacklist. How to install driver and gr module for RTL2832

Here's an image with everything we've discussed so far - XillyBus with the working GNU Radio and the RTL-SDR driver. Go ahead and use it... if you're lazy Pre Compiled image.

We were given a gentle suggestion by Professor Kastner to take a look at an alternative to XillyBus, a version of linux called PetaLinux that is officially supported by Xilinx. We did not end up using it for reasons discussed in the following link. A Little About PetaLinux

A simple test of adding 2 to a number of our choosing to prove that we were able to send/receive data to/from the FPGA. Creating FPGA Image with Xillybus

When using Xillybus, we were seeing speeds that were simply insufficient to keep up with GNU Radio. This details our experiments to try and raise the data rate via Xillybus. HINT: the answer was to send more data at a time. Xillybus Performance Analysis

In order to collect data for our test bench we had to save input/output of the blocks in GNU Radio Companion. Here is a simple python script to go line by line through the data that was dumped to a file in binary and translate it into real numbers you can read. How to convert GR binary sink file to text file

The block we wanted to port into the FPGA was the WBFM, or Wideband FM receiver. This is the analysis from the top layer of python code within GRC all the way down to the individual C++ functions that are called. These are the functions you need to move into HLS in order to create a proper working bitstream which will process the data the same as the GNU Radio Companion block will. WBFM Code Analysis

When looking at the code, we were able to see that quite a few of the parameters (such as filter taps) were created on the fly. However, because we're always using the same filters for FM radio, we were able to hardcode those as well as a few other items in order to simplify the task of coding in HLS. Determining which WBFM Receiver Parameters We Can Hardcode

This is a generalized write-up on how to create new GNU Radio Companion blocks using the built-in tool gr_modtool which comes with GNU Radio. This is what we had to do in order to talk to the FPGA from GRC. How to Create GRC Blocks using gr_modtool

This is the actual code within HLS that calls the functions we ported over in order to process the data correctly. These functions can be found in the source folder. Xillybus Wrapper

This is how we were able to capture test data from the C++ blocks called in GRC so that we were able to test each function individually in HLS. There was some manual cleanup required in order to take numbers out of scientific notation, but it's fairly trivial with something like Excel. Data collection for test bench for WBFM

This is the exact process, using gr_modtool, to make blocks in GNU Radio that will communicate with the FPGA 32 bits at a time. Making GRC blocks for Xillybus

When you get your block working (be it WBFM or something else) and passing your test bench with zero errors, you still have to make RTL. And this is how. Creating IP using vivado HLS

The final process is going to be creating a new bitstream from the RTL you just created. Integrating the bitstream using vivado 2014.1

After this, you should have blocks to communicate with the FPGA, a working set of code in HLS that matches with the test bench data you captured, and IP with the proper xilly_wrapper to take in data and send it to your functions. If you don't know how to do that exactly... well go through the XillyBus tutorials!
